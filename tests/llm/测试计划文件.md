# LLM 连通性测试计划 (LLM Connectivity Test Plan)

## 测试目标

1. **证明正常情况稳定可用**: 当配置正确的 API Key 和网络环境下，LLM 插件能够成功执行健康检查（health_probe）并返回 HEALTHY 状态。
2. **证明异常和特殊情况不会把系统拖入失控**: 当 API Key 错误、网络超时或提供商服务异常时，系统应能捕获并返回正确的错误类型/状态，而不是崩溃。
3. **主动发现未知问题**: 通过真实的探测发现网络延迟或 API 版本不兼容问题。
4. **为稳定性增强、回滚设计、降级设计提供证据**: 验证 DEGRADED 状态的触发（如限流）。

## 验证方法详述

### 1. 模拟健康检查测试 (Unit/Mock Test)

- **准备**: 准备一个 Mock 的 `GoogleGeminiClient`。
- **执行**: 调用 `GeminiProvider.health_probe()`。
- **断言**:
  - 模拟成功时，返回 `PluginHealthStatus.HEALTHY`。
  - 模拟 429 时，返回 `PluginHealthStatus.DEGRADED`。
  - 模拟 500 时，返回 `PluginHealthStatus.UNHEALTHY` (ModelProviderRemoteError)。
  - 模拟 401/403 时，抛出 `ModelProviderHealthError` (wrapped `ModelProviderAuthError`)。
- **区分**: 若 Mock 逻辑报错，则是测试写错；若 Provider 没能正确映射错误，则是产品缺陷。

### 2. 真实连通性测试 (True Connectivity Test)

- **准备**: 检查环境变量 `GEMINI_API_KEY` 是否设置。
- **执行**: `pytest tests/llm/test_llm_connectivity.py` (包含实机调用代码)。
- **断言**: 检查返回状态是否符合预期。
- **失败记录**: 若网络不可达（且外部互联网正常），记录为环境/网关问题。
