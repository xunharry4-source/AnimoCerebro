# AnimoCerebro 项目总结（基于当前代码）

本文档基于当前仓库实际代码、页面入口、后端路由与测试分布整理，重点说明：

更新时间：2026-04-08

## 1. 项目当前定位

当前项目是一个**生产就绪级别**的智能体系统工程，已实现从认知推理到执行监控、从学习到进化、从记忆到反思的完整闭环。核心组成部分包括：

- **`zentex runtime`**：负责运行时、会话管理、九问认知链、工具编排、Transcript 记录等核心能力，已实现完整的九阶段认知循环
- **`web_console`**：基于 FastAPI 的后端控制台接口层，向前端暴露运行态、任务、插件、审计、九问、学习、升级等完整能力
- **`admin-portal`**：基于 React + Vite 的管理前端，提供实时指挥台、九问页面、任务/Agent/插件/MCP 管理、记忆推理、模拟探索等 UI
- **`plugins`**：插件与外部能力层，承载九问插件、感知插件、执行插件、模型 Provider、记忆/模拟/权重/反思等扩展能力
- **`learning`**：G16 学习引擎，支持 DSPy 适配、预算控制、沙箱执行等学习能力
- **`upgrade`**：受控自我进化系统，支持插件演化、证据追踪、审计日志等升级能力
- **`reflection`**：元认知反思系统，支持深度反思、持久化、服务化等反思能力
- **`memory`**：增强记忆系统，支持语义/程序/情景记忆，集成 Kuzu 图数据库后端

从当前代码看，项目已经进入"生产就绪 + 多模块深度联调 + 完整管理台可视化 + 全面测试覆盖"的成熟阶段。

## 2. 当前已实现的核心功能

### 2.1 运行时主链（高度成熟）

`src/zentex/runtime` 已实现项目最核心的完整运行时能力体系：

- **`BrainRuntime`**（运行时大管家）
  - 负责装配运行时依赖，包括所有认知器官和工具注册中心
  - 管理会话实例生命周期，支持多会话并发
  - 维护运行时状态快照，提供统一的状态投影
  - 维护九问状态与冷启动初始化状态，支持事件驱动刷新
  - 集成人工干预控制平面，支持暂停、角色变更等操作
  - 集成工作记忆控制器、时间引擎、活体自我模型引擎等认知器官

- **`BrainSession`**（会话容器）
  - 负责单个会话的连续性状态管理，支持从 transcript 完全恢复
  - 持有最近一轮工作记忆、元认知、模拟结果、决策结果等完整状态
  - 支持 Transcript 回放机制，确保主体连续性
  - 集成九问状态缓存，支持热路径读取
  - 支持身份包插件化，角色包/禁令包/行业经验包可独立加载切换

- **`ThinkLoop`**（九阶段认知执行器）
  - 已实现完整的九阶段认知循环协议，阶段顺序固定不可变
  - 九个阶段：观察→框架化→工作状态刷新→风险检测→模拟→元认知→工具编排→决策综合→巩固
  - 已具备冷启动九问推理入口，支持完整的环境指纹识别
  - 在关键阶段（Phase 2 框架化、Phase 8 决策合成）接入真实 LLM Provider 调用
  - 实现完整的模型调用溯源机制，包含调用方身份、所处阶段、九问驱动来源
  - 支持人工干预守卫机制，在关键阶段可暂停等待人工确认
  - 实现上下文语义翻译层，将内部工程字段转换为模型友好的自然语言
  - 集成进化引擎，在决策后自动推导所有进化画像

- **`BrainTranscriptStore`**（转录存储）
  - 使用 JSONL append-only 方式保存所有运行事件
  - 支持按 `session_id` / `turn_id` 高效读取
  - 为回放、恢复、审计提供唯一事实来源
  - 支持事件监听器机制，可对接增强记忆系统
  - 记录完整的 trace chain，支持 phase 级精准追踪

- **其他核心组件**
  - `WorkingMemoryController`：工作记忆控制器，支持注意力项管理
  - `TemporalEngine`：时间议程引擎，评估时间敏感事项
  - `LivingSelfModelEngine`：活体自我模型引擎，动态更新认知状态画像
  - `MetacognitionController`：元认知控制器，生成推理模式和工具计划
  - `ConflictEngine`：冲突检测引擎，识别认知风险和置信度漂移
  - `NineQuestionExecutor`：九问执行器，支持批量问题执行和脏标记管理
  - `NineQuestionRouter`：九问路由器，发布和管理九问事件
  - `NineQuestionState`：九问状态，缓存问题快照和驱动引用

### 2.4 LLM / Provider 接入（fail-closed 机制）

从 `provider_tools.py`、`model_provider_spec.py`、`web_console/services/llm.py` 以及全局 LLM guard 逻辑看，当前已实现完整的 LLM 接入和防护机制：

- **Provider 工具装配**
  - 默认 LLM Provider 工具自动装配
  - 支持多种 provider 配置（OpenAI 兼容、Gemini 等）
  - Provider 配置检查和验证

- **错误处理与分类**
  - API Key 缺失时 fail-closed，阻止相关操作
  - LLM 调用异常精细分类映射：
    - 认证错误（401）：API Key 或网关鉴权问题
    - 配置错误（503）：缺少必要参数或配置错误
    - 限流（429）：额度用尽或被限流
    - 超时（504）：网络不可达或超时
    - 返回结构错误（502）：模型返回无效结构
    - 远程错误（502）：provider 服务异常

- **Web Console 防护**
  - 在所有写操作前进行 LLM 可用性拦截
  - 对 POST/PUT/PATCH/DELETE 请求进行守卫检查
  - 排除插件管理、人工干预、记忆、升级等不依赖 LLM 的操作
  - 提供详细的错误提示和修复建议

- **溯源机制**
  - 每次模型调用都记录完整的 caller_context
  - 包含 source_module、invocation_phase、question_driver_refs、decision_id
  - 支持 trace_chain 追踪，关联 phase 和 request_id
  - 记录请求和响应的完整 payload

这说明项目已经严格要求"真实 LLM 可用才允许进入相关主链"，而不是伪造 AI 结果，并且提供了完善的错误处理和溯源能力。
