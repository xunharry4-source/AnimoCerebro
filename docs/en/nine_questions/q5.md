# Q5: What Cannot I Do / Safety Rules & Goal Filtering

## Core Mission

Based on upstream facts, long-term prohibitions, role boundaries, asset risk profiles, and historical accident evidence, generate `GlobalSafetyRuleSet`, filter Q4 goal candidates, and bind applicable safety rules to passing candidates.

## Detailed Functional Specifications

- **LLM Requirement**: `LLM_MANDATORY`. Used for semantic-level safety filtering and rule induction.
- **Core Positioning**:
    1. Generate this round's universal `GlobalSafetyRuleSet`.
    2. Filter Q4 goals based on rules (output `passed` / `blocked` / `rewrite_required`).
    3. Bind rule references to passing goals (`CandidateSafetyRuleBinding`).
- **Basis Sources**:
  - Highest priority: Human-set roles, manual authorization boundaries, IdentityKernel hard constraints.
  - Context: Q1 environmental posture, Q2 asset profiles, Q3 effective roles, Q4 goal candidates.
  - Governance experience: Historical failures, accident memories, platform rules, legal compliance requirements.
- **Internal Cognitive Lane Responsibilities**: Bind internal rules (such as no self-rewriting of core modules, no overriding human main roles, etc.).
- **External Execution Lane Responsibilities**: Bind external rules (such as no unauthorized deletion, no bypassing authorization, no externalizing private data, etc.).
- **Rules**: Q5 is only responsible for safety admission and rule binding; it does not generate consequence analysis (Q6) or alternative solutions (Q7).

## Business Examples

### Example A: Internal Cognitive Lane / Engineering Robustness Upgrade

- **Allowed**: Add shadow testing processes, candidate patch verification, test evidence package generation.
- **Prohibited**: Automatically overwrite safety, identity, supervision, audit core modules; prohibit merging directly without confirmation.

### Example B: External Execution Lane / Reddit Public Content Research

- **Allowed**: Read public rules, analyze public hot post structures, manually review drafts.
- **Prohibited**: Batch register or control accounts, vote manipulation, ban evasion, scraping non-public data.

### Example C: External Application Connector / Office Document Processing

- **Prohibited Items**: Prohibit unauthorized reading of files outside allowed_roots; prohibit overwriting original files; prohibit externalizing document content.

## In-Depth Details

### Inputs/Outputs

- **Inputs**:  
  - Objective seed sets from **Q4** (`ObjectiveSeedSet` / `ObjectiveCandidateSet`)  
  - Environmental and self-observation signals from **Q1**, **Q2** (`EnvironmentObjectiveSignalSet`, `SelfObservationObjectiveSignalSet`)  
  - Current role hypothesis (`effective_role`, i.e., Q3 output)  
  - Upstream facts such as human-set roles and identity kernel constraints, long-term prohibitions, asset risk profiles, historical accident records, tool profiles, platform/organizational rules.  

- **Outputs**:  
  - `GlobalSafetyRuleSet`: Comprehensive safety rule set for internal and external lanes.  
  - `Q4ObjectiveSafetyFilterResult`: Filtering results for Q4 goal candidates, marking `passed`, `blocked`, `rewrite_required`.  
  - `CandidateSafetyRuleBinding[]`: Applicable safety rule references bound to passing goals.  
  - All outputs include provenance information for audit chain traceability.

### Key Responsibilities

1. **Generate Total Safety Rules**: Construct `GlobalSafetyRuleSet` based on highest-priority human configurations, IdentityKernel hard constraints, and governance experience.  
2. **Semantic-Level Goal Filtering**: Use live LLM with `LLM_MANDATORY_FOR_SEMANTIC_BOUNDARY_SYNTHESIS_AND_TARGET_FILTERING` to match Q4 goal seeds with safety rules, producing `Q4ObjectiveSafetyFilterResult`.  
3. **Rule Binding**: Generate `CandidateSafetyRuleBinding` for each passing goal, noting applicable internal/external rule tags (such as `no_identity_kernel_write`, `requires_human_authorization`, etc.).

### Security & Compliance

- **Not Responsible for Costs/Consequences**: Q5 only performs safety filtering and rule binding; cost analysis and consequence constraints are completed by **Q6**.  
- **LLM Mandatory**: All semantic deductions must use live LLM, marked `LLM_MANDATORY_FOR_SEMANTIC_BOUNDARY_SYNTHESIS_AND_TARGET_FILTERING`.  
- **Audit Traceability**: Outputs must include `provenance` fields pointing to all input sources (human configurations, IdentityKernel, tool profiles, historical risks, etc.) to support complete audit chains.

## Dual-Lane Differences: Soul Integrity vs. Risk Containment

Q5 has essential differences in its "protection objects" when making safety rulings for internal vs. external lanes:

### 1. Internal Cognitive Lane

- **Protection Core**: **"Soul Integrity"**.
- **Safety Assets**: Non-bypassable constraints of identity kernel (IdentityKernel), memory continuity rules, human supervision channel status.
- **Ruling Focus**:
  - **Prevent Personality Drift**: Does the goal attempt to modify system meta-motivations or core role settings?
  - **Protect Core Memories**: Does the goal involve destructive cleanup of key reflection bases or historical anchors?
  - **Prevent Self-Bypassing**: Does the goal attempt to close audit logs or bypass safety gates?
- **Goal Emphasis**: Ensure every step of evolution does not betray the original "self."

### 2. External Execution Lane

- **Protection Core**: **"Risk Containment"**.
- **Safety Assets**: Execution permission matrix (Rights Matrix), external API red lines, organizational compliance policies, physical resource quotas.
- **Ruling Focus**:
  - **Prevent Unauthorized Operations**: Does the goal involve unauthorized directory reading or remote overwriting?
  - **Data Privacy Protection**: Does the goal involve non-desensitized externalization of sensitive information?
  - **Physical Side Effect Control**: Will the goal consume excessive bandwidth, trigger API circuit breakers, or cause irreversible impacts on external systems?
- **Goal Emphasis**: Ensure every execution step remains within the physical and compliance "safety cage."

## Classification Audit Strategy: Dynamically Adjust Safety Intensity Based on Value Dimensions

Q5 no longer uses a single "one-size-fits-all" filtering method, but adopts differentiated safety audit logic based on the six value dimensions output by Q4:

### 1. Business Delivery (Task Delivery)

- **Intensity**: **Highest**.
- **Focus**: Strictly compare physical red lines and user authorization boundaries. These actions directly intervene in reality and carry the highest risk.

### 2. Cognitive Evolution (Self-Evolution)

- **Intensity**: **Medium-High**.
- **Focus**: **"Soul Integrity"**. Prevent accidental modification of meta-motivations or core roles (IdentityKernel) during evolution.

### 3. Asset Governance (Asset Governance)

- **Intensity**: **Medium-High**.
- **Focus**: **"Operational Risk"**. Ensure that optimizing tools (such as plugins, private MCP) does not cause systemic paralysis or code damage.

### 4. Strategic Discovery & Advisory (Strategic Discovery & Advisory)

- **Intensity**: **Lowest/Advisory**.
- **Focus**: **"Discovery Is Compliance"**. Advisory behavior identifying uncontrollable external defects should be allowed by default, only marked as `recommendation_only`.

### 5. Preemptive Guardianship (Preemptive Guardianship)

- **Intensity**: **Low/Isolation**.
- **Focus**: **"Read-Only & Non-Intervention"**. Monitoring tasks should be forcibly restricted to read-only, strictly prohibiting write operations during guardianship.

### 6. Knowledge Consolidation (Knowledge Consolidation)

- **Intensity**: **Medium**.
- **Focus**: **"Privacy & Consistency"**. Ensure precipitated knowledge does not contain undesensitized sensitive data.
