# Q9: How Exactly To Implement / Formulate Implementation Plans

## Core Mission
Based on Q8's final task and goal combinations, generate implementation plans, step drafts, input/output contracts, verification methods, and failure recovery schemes. Q9 is still at the cognitive layer, not the execution layer, and does not produce physical work orders.

## Detailed Functional Specifications
- **LLM Requirement**: `LLM_MANDATORY`. Used to generate plan drafts and verification contracts.
- **Internal Cognitive Lane Returns**: `InternalImplementationPlanDraft`, containing cognitive steps, target modules, contracts, verification steps, rollback steps, etc.
- **External Execution Lane Returns**: `ExternalImplementationPlanDraft`, containing action intent drafts, execution steps (Agent/CLI/etc.), permission requirements, receipt requirements, timeout strategies, etc.
- **Verification Contract Requirements**: Each step must include verification methods, prioritizing objective physical evidence (file hashes, IDs, read-after-write, etc.). Prohibit using only "looks successful" as the standard.
- **Boundary Rules**:
    - Must not generate task_id, subtask_id, or resource locks.
    - Must not directly invoke tools.
    - Output results enter `CognitiveAgendaDraft` or `ActionIntentDraft`, instantiated by G31A Task Center.

## Business Examples

    2. Design ShadowTestingRunner interface.
    3. Apply patches in sandbox and run lint/test.
    4. Generate EvidenceBundle.
- **Verification Methods**: Real execution exit_code, test report summaries, candidate area diffs.

    4. Record all source citations and audit traces.
- **Verification Methods**: Public source citations exist, rule cards non-empty, audit traces complete.

### Example C: External Application Connector / Office Document Processing
- **Plan Content**: Includes read_document, summarize, export_pdf, read-after-export verification, EvidenceBundle generation, and failure recovery logic.

## In-Depth Details

### Inputs/Outputs

- **Inputs**:
  - Final task and goal combinations from **Q8** (`FinalObjectiveProfile`)
  - Safety rule packages from **Q5** (`GlobalSafetyRuleSet` + filtering results)
  - Plan constraints from **Q6** (`PlanConstraintSet`)
  - Creative exploration sets from **Q7** (`CreativePossibilitySet`)
  - Current role hypothesis (`effective_role`, i.e., Q3 output)
  - Environmental and self-observation signals (`EnvironmentObjectiveSignalSet`, `SelfObservationObjectiveSignalSet`)

- **Outputs**:
  - `ImplementationPlanDraft`: Internal `InternalImplementationPlanDraft` and external `ExternalImplementationPlanDraft`, containing step sequences, input/output contracts, verification methods, failure recovery schemes, and `provenance` information for downstream task execution engines (G31A) to instantiate specific `task_id`.

### Key Responsibilities

1. **Generate Implementation Plan Drafts**: Break down goals into ordered execution steps based on `FinalObjectiveProfile`, clarifying preconditions, dependencies, and outputs for each step.
2. **Define Contracts**: Provide **input/output schema**, **verification methods** (file hash, read-after-write, audit trace, etc.) for each step to ensure observability and traceability.
3. **Safety Compliance**: Embed safety tags from `GlobalSafetyRuleSet` in drafts, prohibiting any unauthorized writes, resource lock creation, or direct external tool invocation.
4. **Failure Recovery**: Formulate rollback or degradation strategies for each step (such as sandbox rollback, retry counts, manual intervention), marking `rollback_step` and `fallback_logic` in `ImplementationPlanDraft`.
5. **Audit Traceability**: All steps must record `provenance`, pointing to upstream signals (Q4, Q5, Q6, Q7) that produced the step and used LLM template IDs for full-chain auditing.

### Security & Compliance

- **Do Not Generate Entity Task IDs**: Q9 only outputs plan drafts; actual `task_id`, `subtask_id`, and resource locks are created at the physical level by **G31A Task Center**.
- **Follow Safety Rules**: All steps must comply with prohibitions in `GlobalSafetyRuleSet`; if conflicts arise, mark as `blocked` or `requires_human_review` in drafts.
- **Audit Information**: `ImplementationPlanDraft` must include `provenance` lists, recording `Q4ObjectiveSeedRefs`, `Q5SafetyRuleRefs`, `Q6ConstraintRefs`, `Q7PossibilityRefs`, etc., ensuring traceability.

## Dual-Lane Differences: Cognitive Consistency vs. Artifact Verification Contracts

Q9 has significant differences in its "contract forms" and "success definitions" when formulating implementation plans for internal vs. external lanes:

### 1. Internal Cognitive Lane
- **Plan Core**: **"Cognitive Consistency"**.
- **Implementation Assets**: Cognitive primitives (such as read/write instructions for specific storage slots), reflection hooks, internal state snapshots.
- **Plan Focus**:
  - **Atomic Operations**: Ensure each cognitive step (such as modifying a reflection strategy) is atomic and can be rolled back without loss on failure.
  - **Causal Tracing**: Each step must clarify its impact on the "brain" state and record complete causal chains.
  - **State Alignment**: Post-execution internal state must be consistent with the "evolved state" expected by Q1/Q2.
- **Goal Emphasis**: Ensure **logical rigor and traceability** during system self-restructuring.

### 2. External Execution Lane
- **Plan Core**: **"Artifact Verification"**.
- **Implementation Assets**: Physical tool invocations (CLI, API), file handles, network ports, external storage space.
- **Plan Focus**:
  - **Strong Verification Contracts**: Each step must include physical evidence validation (such as file hash verification, read-after-write verification).
  - **Exception Defense**: Design timeout, retry, and physical environment recovery schemes (such as deleting temporary files) for each external action.
  - **Audit Closure**: Ensure artifacts produced by each action are correctly indexed and stored in the audit system.
- **Goal Emphasis**: Ensure **execution robustness and result verifiability** when the system intervenes in the external world.
