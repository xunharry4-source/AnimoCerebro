## 环境变量配置

### 配置文件位置

- **示例文件**: `.env.example`
- **本地配置**: `.env`（不提交到版本控制）

### 设置步骤

```bash
# 1. 复制示例文件
cp .env.example .env

# 2. 编辑 .env 文件，填入实际的 API Key
vim .env
```

### 主要配置项

```env
# ============================================
# Local OpenAI-compatible gateway
# ============================================
# 本地 OpenAI 兼容网关配置（推荐用于开发）
OPENAI_COMPAT_BASE_URL=http://localhost:8317/v1
OPENAI_COMPAT_API_KEY=your-api-key-1
OPENAI_COMPAT_MODEL=gemini-3-flash(auto)

# ============================================
# Direct provider keys
# ============================================
# 直接使用各 Provider 的 API Key
OPENAI_API_KEY=sk-...              # OpenAI API Key
GEMINI_API_KEY=AIza...             # Google Gemini API Key
ANTHROPIC_API_KEY=sk-ant-...       # Anthropic Claude API Key
```

### 配置说明

#### 1. OpenAI 兼容网关（推荐开发使用）

适用于本地部署的 LLM 网关（如 Ollama、LiteLLM、OneAPI 等）：

```env
OPENAI_COMPAT_BASE_URL=http://localhost:8317/v1
OPENAI_COMPAT_API_KEY=your-api-key-1
OPENAI_COMPAT_MODEL=gemini-3-flash(auto)
```

**优势**：
- 统一管理多个模型提供商
- 支持模型路由和负载均衡
- 便于本地开发和测试

#### 2. 直接 Provider 配置

直接使用各 Provider 的官方 API：

```env
OPENAI_API_KEY=sk-...              # OpenAI
GEMINI_API_KEY=AIza...             # Google
ANTHROPIC_API_KEY=sk-ant-...       # Anthropic
```

**适用场景**：
- 生产环境
- 需要直接使用特定 Provider 的高级功能
- 测试特定 Provider 的性能

## LLM Provider 配置

### 配置文件位置

`config/provider_tools.yml`

### 配置结构

```yaml
providers:
  <provider_id>:
    provider_name: <名称>
    api_base: <API 基础 URL>
    api_key_env: <环境变量名>
    default_model: <默认模型>
    timeout_seconds: <超时时间>
```

### 完整配置示例

```yaml
providers:
  # OpenAI 兼容网关
  openai_compat:
    provider_name: openai_compat
    api_base: http://localhost:8317/v1
    api_key_env: your-api-key-1
    default_model: gemini-3-flash(auto)
    timeout_seconds: 30

  # OpenAI 官方 API
  openai:
    provider_name: openai
    api_base: https://api.openai.com/v1
    api_key_env: OPENAI_API_KEY
    default_model: gpt-4.1-mini
    timeout_seconds: 30

  # ChatGPT（与 OpenAI 相同）
  chatgpt:
    provider_name: chatgpt
    api_base: https://api.openai.com/v1
    api_key_env: OPENAI_API_KEY
    default_model: gpt-4.1-mini
    timeout_seconds: 30

  # Google Gemini
  gemini:
    provider_name: gemini
    api_base: https://generativelanguage.googleapis.com/v1beta
    api_key_env: GEMINI_API_KEY
    default_model: gemini-1.5-pro
    timeout_seconds: 30

  # Anthropic Claude
  claude:
    provider_name: claude
    api_base: https://api.anthropic.com/v1
    api_key_env: ANTHROPIC_API_KEY
    default_model: claude-3-5-sonnet-latest
    timeout_seconds: 30
```

### 添加新 Provider

1. 在 `config/provider_tools.yml` 中添加配置：

```yaml
providers:
  my_custom_provider:
    provider_name: my_custom_provider
    api_base: https://api.example.com/v1
    api_key_env: MY_CUSTOM_API_KEY
    default_model: model-name
    timeout_seconds: 30
```

2. 在 `.env` 中添加对应的 API Key：

```env
MY_CUSTOM_API_KEY=your-api-key
```

3. 重启服务使配置生效：

```bash
make restart-dev
```

### Provider 选择策略

系统会根据以下优先级选择 Provider：

1. **显式指定**：代码中明确指定的 provider_id
2. **默认 Provider**：配置中的第一个 provider
3. **Fallback**：如果主 Provider 失败，尝试其他可用的 Provider

## 端口配置

### 默认端口

- **后端**: 8000
- **前端**: 5173

### 自定义端口

#### 方法 1: 环境变量（推荐）

```bash
# 临时修改
BACKEND_PORT=8001 FRONTEND_PORT=5174 make dev

# 永久修改：在 .env 中添加
echo "BACKEND_PORT=8001" >> .env
echo "FRONTEND_PORT=5174" >> .env
```

#### 方法 2: 修改脚本

编辑 `scripts/dev_all.sh`，修改默认值：

```bash
BACKEND_PORT="${BACKEND_PORT:-8001}"  # 改为 8001
FRONTEND_PORT="${FRONTEND_PORT:-5174}"  # 改为 5174
```

### 端口冲突处理

如果端口被占用，使用重启命令：

```bash
# 一键清理并重启
make restart-dev

# 或切换到其他端口
BACKEND_PORT=8001 FRONTEND_PORT=5174 make restart-dev
```

## WebSocket 配置

### WebSocket 实现

项目统一使用 `websockets-sansio` 实现：

```bash
# 通过环境变量配置
export ZENTEX_WS_IMPLEMENTATION=websockets-sansio
```

### 可用实现

- `websockets-sansio`（默认，推荐）
- `wsproto`

### 配置位置

在启动脚本中自动设置：

```bash
# scripts/dev_all.sh
WS_IMPLEMENTATION="${WS_IMPLEMENTATION:-websockets-sansio}"
export ZENTEX_WS_IMPLEMENTATION="${WS_IMPLEMENTATION}"
```

## 数据库配置

### Kuzu 图数据库

记忆系统使用 Kuzu 图数据库作为后端。

#### 数据库位置

```
.zentex/kuzu_db/
```

#### 配置项

Kuzu 数据库的配置通常在代码中硬编码，如需修改：

```python
# src/zentex/memory/kuzu_backend.py
db_path = ".zentex/kuzu_db"
```

#### 清理数据库

```bash
# 删除数据库（谨慎操作！）
rm -rf .zentex/kuzu_db
```

### SQLite 缓存

状态缓存使用 SQLite：

```
app_data/cache/state_v2/cache.db
```

## 日志配置

### 日志级别

通过环境变量配置：

```bash
export LOG_LEVEL=DEBUG  # DEBUG, INFO, WARNING, ERROR, CRITICAL
```

### 日志位置

- **后端日志**: 终端实时输出
- **Transcript 日志**: `.zentex/runtime/*.jsonl`

### Transcript 文件

```
.zentex/runtime/
├── web_console_transcript.jsonl   # Web Console 事件流
├── brain_transcript.jsonl         # Brain 运行事件
├── enhanced_episodic.jsonl        # 情景记忆
├── enhanced_semantic.jsonl        # 语义记忆
├── enhanced_procedural.jsonl      # 程序记忆
└── enhanced_management.jsonl      # 管理记忆
```

### 清理大型日志文件

重启脚本会自动清理大于 1MB 的日志文件：

```bash
make restart-dev
```

或手动清理：

```bash
# 清空所有 transcript 文件
for f in .zentex/runtime/*.jsonl; do > "$f"; done
```

## 高级配置

### Python 路径

确保 PYTHONPATH 包含 src 目录：

```bash
export PYTHONPATH=src
```

启动脚本会自动设置。

### 虚拟环境

推荐使用 Python 虚拟环境：

```bash
# 创建虚拟环境
python3 -m venv .venv

# 激活虚拟环境
source .venv/bin/activate  # macOS/Linux
# 或
.venv\Scripts\activate     # Windows

# 安装依赖
pip install -r requirements.txt -r requirements-dev.txt
```

Makefile 会自动处理：

```bash
make backend-install
```

### Node.js 版本

推荐使用 Node.js 18+：

```bash
node --version  # 检查版本
```

### CORS 配置

前后端分离开发时，CORS 已自动配置：

```python
# src/zentex/web_console/app.py
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 开发环境允许所有来源
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

生产环境应限制允许的源：

```python
allow_origins=["https://your-domain.com"]
```

## 配置验证

### 检查配置是否生效

```bash
# 1. 启动服务
make dev

# 2. 检查后端健康状态
curl http://127.0.0.1:8000/api/web/overview

# 3. 检查 LLM Provider 配置
curl http://127.0.0.1:8000/api/web/model_feature_tests/providers

# 4. 查看日志输出
# 启动时会显示使用的配置
```

### 常见问题

#### 1. API Key 未配置

**症状**: LLM 调用失败，提示缺少 API Key

**解决**:
```bash
# 检查 .env 文件
cat .env

# 确保设置了正确的 API Key
OPENAI_API_KEY=sk-...
```

#### 2. 端口被占用

**症状**: 启动失败，提示端口已被占用

**解决**:
```bash
# 使用重启命令
make restart-dev

# 或切换端口
BACKEND_PORT=8001 make dev
```

#### 3. 依赖缺失

**症状**: 导入错误，模块找不到

**解决**:
```bash
# 重新安装依赖
make backend-install
make frontend-install
```

#### 4. 数据库损坏

**症状**: 记忆系统报错

**解决**:
```bash
# 备份并重建数据库
mv .zentex/kuzu_db .zentex/kuzu_db.backup
make restart-dev
```
