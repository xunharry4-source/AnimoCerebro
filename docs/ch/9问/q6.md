# Q6: 这样做的代价是什么 / 计划限制与后果约束

## 核心使命
为实现目标将要生成的计划加条件、加后果约束、加合理性约束，防止为了达成目标而不计后果。Q6 的输出对后续的 Q8/Q9 是强制约束。

## 详细功能规范
- **LLM 要求**：`LLM_MANDATORY`。用于推演代价、影响、不合理项和坏后果。
- **内部认知轨返回**：`InternalPlanConstraintSet`，包含认知成本、记忆影响、价值漂移风险、策略污染风险、回滚要求、暂停/停止条件等。
- **外部执行轨返回**：`ExternalPlanConstraintSet`，包含物理副作用、影响半径、数据暴露风险、成本失控风险、证据要求、回滚难度等。
- **任务合理性职责**：指出目标或计划设想是否合理，是否需要拆分/合并，是否有更简单低风险路径。
- **规则**：
    - Q6 不决定“能不能做”（那是 Q5 的职责）。
    - Q6 不写具体计划（那是 Q8/Q9 的职责）。
    - 约束必须被 Q8 和 Q9 强制消费。

## 业务示例

### 示例 A：内部认知轨 / 工程鲁棒性升级
- **注意事项**：必须先只读分析现有补丁流程；必须在隔离目录或沙箱运行；必须记录详细测试日志。
- **影响与后果**：沙箱误配置会拖慢开发；测试覆盖不足会产生伪安全感。
- **限制条件**：不可触碰保护模块；必须有 rollback checkpoint 和 EvidenceBundle。

### 示例 B：外部执行轨 / Reddit 公开内容研究
- **注意事项**：只读公开数据；必须遵守 robots.txt 和 API 规则；不得自动发帖。
- **影响与后果**：公开内容分析可能带来偏差；过度模板化会降低内容质量。
- **限制条件**：所有输出必须进入人工审核；需要速率限制和审计 trace。

### 示例 C：外部应用连接器 / Office 文档处理
- **约束项**：要求先备份或只读读取；导出到指定目录；必须执行 hash/mtime 验证；不得仅相信 success=true。
## 深入细节

### 输入/输出

- **输入**：  
  - 来自 **Q4** 的目标种子集合 (`ObjectiveSeedSet` / `ObjectiveCandidateSet`)  
  - 来自 **Q5** 的安全规则包 (`GlobalSafetyRuleSet` + 过滤结果)  
  - 当前角色假设 (`effective_role`，即 Q3 输出)  
  - 环境与自我观察信号 (`EnvironmentObjectiveSignalSet`, `SelfObservationObjectiveSignalSet`)  
  - 能力边界证据 (`CapabilityBoundaryEvidenceRefs`)  

- **输出**：  
  - `PlanConstraintSet`：内部 `InternalPlanConstraintSet` 与外部 `ExternalPlanConstraintSet`，分别包含代价评估、后果约束、合理性判断、暂停/停止条件、回滚要求等。  
  - 约束说明文档，用于后续 **Q8**/ **Q9** 实现计划的强制消费。

### 关键职责

- **代价评估**：量化资源消耗、计算成本、时间延迟、风险暴露等。  
- **后果约束**：明确可能的负面影响（数据泄露、系统不稳定、法律合规风险），并提出缓解措施。  
- **合理性判断**：判断目标是否值得执行，是否需要拆分、合并或重新评估。  
- **安全合规**：遵循 **Q5** 输出的 `GlobalSafetyRuleSet`，在约束中引用相应的安全标签（`requires_human_authorization`、`no_unapproved_external_write` 等），但不自行进行安全过滤。

### 安全与合规

Q6 本身 **不决定能否执行**（那是 Q5），只负责在已通过安全过滤的目标上提供**约束和代价**。  
- 所有约束必须在 **Q8**/ **Q9** 中被强制采纳；若下游阶段忽略约束，将触发审计警报。  
- 约束产物必须携带 provenance 信息，指向输入来源，以便审计链追溯。

### 实现示例

#### 内部示例 – 影子测试升级约束

```text
目标种子：升级影子测试 Shadow Testing 能力。
约束输出：
- 代价：需要额外 2GB 内存 + 5 分钟 sandbox 执行时间。
- 后果：若 sandbox 失败，将导致 Worker 重启风险。
- 合规标签：`no_identity_kernel_write`、`requires_human_authorization`。
- 回滚要求：提供已验证的回滚补丁，并记录 hash。
```

#### 外部示例 – Reddit 内容质量分析约束

```text
目标种子：部署内容质量分析与草稿建议能力。
约束输出：
- 代价：每天最多 10,000 次 API 调用，需配额审批。
- 后果：若未遵守 robots.txt 可能触发平台封禁。
- 合规标签：`no_unapproved_external_write`、`requires_cloud_audit_when_physical_params_known`。
- 暂停条件：检测到超过 0.1% 错误率时自动暂停。
```

## 双轨差异：认知可持续性 vs 资源投入产出比

Q6 在评估代价时，内外轨的“计费方式”与“风险模型”完全不同：

### 1. 内部认知轨 (Internal Lane)
- **评估核心**：**“系统可持续性 (Systemic Sustainability)”**。
- **代价资产**：认知预算（Token 消耗、推理时长）、记忆槽位占用、逻辑一致性评分。
- **约束重点**：
  - **防止认知过载**：该目标是否会占用过多上下文，导致系统无法处理并发任务？
  - **记忆熵控**：执行该目标是否会产生大量低价值记忆碎片，导致检索退化？
  - **策略稳定性**：新的策略补丁是否会与现有逻辑冲突，导致“逻辑中毒”？
- **目标侧重**：确保系统在进化过程中不因代价过高而崩溃或陷入混乱。

### 2. 外部执行轨 (External Lane)
- **评估核心**：**投入产出比 (ROI) 与资源效率**。
- **代价资产**：财务预算（云服务费用、API 扣费）、网络带宽、物理存储、操作员的人工确认时间。
- **约束重点**：
  - **经济性审计**：执行该目标所需的 API 费用是否超过其带来的业务价值？
  - **影响半径管控**：该操作是否会波及不相关的外部系统，导致“大面积故障”？
  - **性能基线**：操作是否会触发外部系统的 Rate Limit，导致其他核心业务受阻？
- **目标侧重**：确保每一份外部资源的投入都能换取可量化的现实价值产出。

## 分类代价评估策略：差异化的资源与后果权衡

Q6 在计算“代价”时，会根据目标的六维分类采用不同的权衡模型：

### 1. 业务交付类 (Task Delivery)
- **核心代价**：真金白银的成本（Token, API 扣费）、交付时效、物理资源消耗。
- **后果约束**：侧重于“任务失败”的补救成本与环境副作用。

### 2. 认知进化类 (Self-Evolution)
- **核心代价**：上下文空间（Context Window）的永久性占用、逻辑复杂度增加。
- **后果约束**：侧重于“逻辑污染”与“偏见引入”的长期风险。

### 3. 资产治理类 (Asset Governance)
- **核心代价**：测试时间、系统停机时间（若涉及热重载）。
- **后果约束**：侧重于“把原本好用的工具修坏”的可靠性风险。

### 4. 战略发现与建议类 (Strategic Discovery & Advisory)
- **核心代价**：**极低 (Insignificant)**。仅消耗少量推理 Token。
- **后果约束**：几乎无物理后果。主要约束是“建议的准确性与专业度”。

### 5. 预防性守护类 (Preemptive Guardianship)
- **核心代价**：**持续性开销**（心跳检查、后台轮询导致的持续 Token 消耗）。
- **后果约束**：侧重于“虚假警报”导致的认知噪音。

### 6. 知识固化类 (Knowledge Consolidation)
- **核心代价**：长期存储空间、检索算法的性能衰减。
- **后果约束**：侧重于“无效知识堆积”导致的检索噪声（Entropy）。
