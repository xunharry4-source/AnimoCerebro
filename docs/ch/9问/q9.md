# Q9: 具体怎么实现 / 制定实现计划

## 核心使命
根据 Q8 的最终任务与目标组合，生成实现计划、步骤草案、输入输出契约、验证方式和失败恢复方案。Q9 仍然是认知层，不是执行层，不产生物理工单。

## 详细功能规范
- **LLM 要求**：`LLM_MANDATORY`。用于生成计划草案和验证契约。
- **内部认知轨返回**：`InternalImplementationPlanDraft`，包含认知步骤、目标模块、契约、验证步、回滚步等。
- **外部执行轨返回**：`ExternalImplementationPlanDraft`，包含动作意图草案、执行步骤（Agent/CLI等）、权限需求、回执要求、超时策略等。
- **验证契约要求**：每个步骤必须包含验证方法，且优先使用客观物理证据（文件哈希、ID、read-after-write 等）。禁止仅以“看起来成功”作为标准。
- **边界规则**：
    - 不得生成 task_id、subtask_id 或资源锁。
    - 不得直接调用工具。
    - 输出结果进入 `CognitiveAgendaDraft` 或 `ActionIntentDraft`，由 G31A 任务中心实例化。

## 业务示例

### 示例 A：内部认知轨 / 工程鲁棒性升级
- **实现计划草案**：
    1. 梳理补丁写入路径。
    2. 设计 ShadowTestingRunner 接口。
    3. 沙盒应用补丁并运行 lint/test。
    4. 生成 EvidenceBundle。
- **验证方式**：真实执行 exit_code、测试报告摘要、候选区 diff。

### 示例 B：外部执行轨 / Reddit 公开内容研究
- **实现计划草案**：
    1. 抓取公开规则及热帖摘要。
    2. 生成规则卡片和质量维度。
    3. 生成内容建议并标记 `human_review_required`。
    4. 记录所有来源引用及审计 trace。
- **验证方式**：公开来源引用存在、规则卡片非空、审计 trace 完整。

### 示例 C：外部应用连接器 / Office 文档处理
- **计划内容**：包含 read_document、summarize、export_pdf、read-after-export 验证、EvidenceBundle 生成和失败恢复逻辑。
## 深入细节

### 输入/输出

- **输入**：
  - 来自 **Q8** 的最终任务与目标组合 (`FinalObjectiveProfile`)
  - 来自 **Q5** 的安全规则包 (`GlobalSafetyRuleSet` + 过滤结果)
  - 来自 **Q6** 的计划约束 (`PlanConstraintSet`)
  - 来自 **Q7** 的创造性探索集合 (`CreativePossibilitySet`)
  - 当前角色假设 (`effective_role`，即 Q3 输出)
  - 环境与自我观察信号 (`EnvironmentObjectiveSignalSet`, `SelfObservationObjectiveSignalSet`)

- **输出**：
  - `ImplementationPlanDraft`：内部 `InternalImplementationPlanDraft` 与外部 `ExternalImplementationPlanDraft`，包含步骤序列、输入输出契约、验证方式、失败恢复方案以及 `provenance` 信息，供下游任务执行引擎（G31A）实例化具体 `task_id`。

### 关键职责

1. **生成实现计划草案**：根据 `FinalObjectiveProfile` 将目标拆解为有序的执行步骤，明确每一步的前置条件、依赖关系和产出。
2. **定义契约**：为每一步提供 **输入/输出 schema**、**验证方法**（文件 hash、read‑after‑write、审计 trace 等），确保可观测且可回溯。
3. **安全合规**：在草案中嵌入 `GlobalSafetyRuleSet` 的安全标签，禁止任何未授权的写入、资源锁创建或直接调用外部工具。
4. **失败恢复**：为每一步制定回滚或降级策略（如 sandbox 回滚、重试次数、人工介入），并在 `ImplementationPlanDraft` 中标记 `rollback_step` 与 `fallback_logic`。
5. **审计追溯**：所有步骤必须记录 `provenance`，指向产生该步骤的上游信号（Q4、Q5、Q6、Q7）以及使用的 LLM 模板 ID，以便全链路审计。

### 安全与合规

- **不生成实体任务 ID**：Q9 仅输出计划草案，实际 `task_id`、`subtask_id` 与资源锁由 **G31A 任务中心** 在物理层面创建。
- **遵循安全规则**：所有步骤必须符合 `GlobalSafetyRuleSet` 中的禁止项，若冲突则在草案中标记为 `blocked` 或 `requires_human_review`。
- **审计信息**：`ImplementationPlanDraft` 必须包含 `provenance` 列表，记录 `Q4ObjectiveSeedRefs`, `Q5SafetyRuleRefs`, `Q6ConstraintRefs`, `Q7PossibilityRefs` 等，确保可追溯。

### 实现示例

#### 示例 A – 内部认知轨 / 影子测试升级实现计划

```text
步骤 1: 沙箱运行候选补丁
  输入: shadow_test_candidate_patch
  输出: sandbox_execution_result
  验证: exit_code==0 && evidence_hash匹配
  安全标签: no_identity_kernel_write, requires_human_authorization
  回滚: 若失败，恢复原始代码基线
步骤 2: 生成 EvidenceBundle
  输入: sandbox_execution_result
  输出: evidence_bundle
  验证: hash(evidence_bundle) 与预期 schema 对齐
  回滚: 无（纯记录）
步骤 3: 提交至代码审计系统
  输入: evidence_bundle
  输出: audit_task_id
  验证: audit_passed == true
  回滚: 若 audit 失败，停止后续部署
```

#### 示例 B – 外部执行轨 / Reddit 内容质量分析实现计划

```text
步骤 1: 拉取 Reddit 热帖（只读）
  输入: subreddit_api_credentials
  输出: raw_post_data
  验证: response_status==200 && 数据完整性 hash
  安全标签: no_unapproved_external_write, rate_limit_10k_per_day
  回滚: 超过速率限制时暂停 5 分钟
步骤 2: 生成内容质量评分模型
  输入: raw_post_data
  输出: quality_score_report
  验证: 模型输出符合 schema, confidence>0.8
  安全标签: requires_cloud_audit_when_physical_params_known
  回滚: 若 confidence 低于阈值，标记为需人工复审
步骤 3: 生成人工审核草稿并记录审计 trace
  输入: quality_score_report
  输出: review_draft + audit_trace
  验证: audit_trace 完整性检查
  回滚: 若缺失 trace，重新生成并报警
```

## 双轨差异：认知一致性 vs 产物验证契约

Q9 在制定实现计划时，内外轨的“契约形式”与“成功定义”存在显著差异：

### 1. 内部认知轨 (Internal Lane)
- **计划核心**：**“认知一致性 (Cognitive Consistency)”**。
- **实施资产**：认知原语（如：对特定存储槽位的 read/write 指令）、反思钩子、内部状态快照。
- **计划重点**：
  - **原子性操作**：确保每个认知步骤（如：修改某个反思策略）都是原子的，失败时可无损回滚。
  - **因果追溯**：每一步必须明确其对“大脑”状态的影响，并记录完整的因果链。
  - **状态对齐**：执行后的内部状态必须与 Q1/Q2 预期的“进化后状态”保持一致。
- **目标侧重**：确保系统在自我重构过程中的**逻辑严密性与可追溯性**。

### 2. 外部执行轨 (External Lane)
- **计划核心**：**“产物验证契约 (Artifact Verification)”**。
- **实施资产**：物理工具调用（CLI、API）、文件句柄、网络端口、外部存储空间。
- **计划重点**：
  - **强验证契约**：每一步必须包含物理证据校验（如：文件 Hash 校验、read-after-write 验证）。
  - **异常防御**：为每个外部动作设计超时、重试及物理环境恢复方案（如：删除临时文件）。
  - **审计闭环**：确保每个动作产生的 Artifact 都被正确索引并存入审计系统。
- **目标侧重**：确保系统在干预外部世界时的**执行稳健性与结果可验证性**。

以上深度细节确保 Q9 不仅是一个“作业指导书”，更是系统将战略转化为**高可靠执行指令**的关键环节。
