# Zentex 九问认知审计架构重构测试计划

## 1. 测试目标 (固定 4 项)

1. **证明正常情况稳定可用**: 验证 Q1-Q9 插件在持有 ModelProvider 和 TranscriptStore 的正常上下文中，能通过 `_shared.py` 逻辑完成可追溯的 LLM 调用。
2. **证明异常和特殊情况不会把系统拖入失控**: 证明当缺失模型供应商或日志存储时，系统能立即通过 `require_*` 助手触发 fail-closed 异常，防止静默失败。
3. **主动发现未知问题**: 通过全链路 trace_id 追踪，检查是否存在认知阶段间的上下文泄漏或 trace 路径中断。
4. **为稳定性增强、回滚设计、降级设计提供证据**: 确保所有认知推演均有 `MODEL_PROVIDER_INVOKED` / `COMPLETED` / `FAILED` 物理日志，满足 G31A 合规性。

## 2. 验证方法详述

### A. 认知插件加载与依赖注入测试
*   **物理准备**: 启动 Zentex Runtime 环境，注入模拟 context。
*   **执行**: 调用 `Q1-Q9.run_tool()` 或 `execute()`。
*   **断言点**: 检查是否正确导入 `_shared.py`，是否能成功通过 context 提取 `ModelProviderSpec`。
*   **缺陷判定**: 若报告 `ImportError` 或 `AttributeError`，判定为重构代码污染；若报告 `LLM MANDATORY` 异常，判定为环境缺失（预期行为）。

### B. G31A 溯源日志物理写入验证
*   **准备**: 配置有效的 `BrainTranscriptStore`。
*   **执行**: 运行全量九问推演流程。
*   **副作用检查**: 查看 `transcript` 存储，搜索包含 `nine_question_` 阶段标识的 entry。
*   **断言**: 每一问必须至少产生 2 条日志（开启与完成/失败），且 `trace_id` 必须全链路一致。

### C. 边界合规与失败路径测试
*   **特殊情况**: 禁用 ModelProvider。
*   **执行**: 触发认知审计。
*   **结果**: 必须返回 `RuntimeError` 且日志中记录 `MODEL_PROVIDER_FAILED`（若已执行到调用层）。
